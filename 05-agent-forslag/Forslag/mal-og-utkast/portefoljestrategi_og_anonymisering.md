# Porteføljestrategi og anonymiseringsrutine — Syntax & Flow
> Hvordan bygge en slagkraftig portefølje samtidig som GDPR og personvern ivaretas.

---

## 1. Hvilke prosjekter publiseres?

| Kriterier for publisering | Må være oppfylt |
|---------------------------|-----------------|
| Samtykke fra kunde | ✅ |
| Samtykke fra alle studenter i teamet | ✅ |
| Ingen forretningshemmeligheter | ✅ |
| Ingen personopplysninger (anonymisert) | ✅ |
| Kvalitetsgodkjenning av mentor | ✅ |
| Prosjektet er fullført (ikke avbrutt) | Anbefalt |

---

## 2. Anonymiseringsrutine

Før publisering skal følgende fjernes eller erstattes:

| Type data | Tiltak |
|-----------|--------|
| Kundens navn | Erstatt med generisk beskrivelse: «en lokal dagligvarebutikk i Halden» |
| Ansattes navn | Fjern |
| E-postadresser | Fjern |
| Telefonnumre | Fjern |
| Adresser | Erstatt med by/område |
| Spesifikke økonomiske tall | Avrund eller bruk relative størrelser (f.eks. «under 100 000 kr») |
| System-passord / API-nøkler | Fjern |
| Skjermbilder med personopplysninger | Sensurer (piksler over) |
| Kode med personopplysninger | Fjern eller erstatt med testdata |

---

## 3. Porteføljestrategi

### Hva skal porteføljen vise?
- **Bredde:** IT, design OG økonomi — ikke bare teknologi
- **Prosess:** Vis frem hvordan teamet jobbet (innsikt → design → utvikling → leveranse)
- **Resultat:** Hva ble oppnådd for kunden?
- **Læring:** Hva lærte studentene?

### Mal for case study

```
Prosjekttittel: [kort beskrivende tittel]
Oppdragsgiver: [anonymisert beskrivelse]
Team: [antall] studenter (IT, design, økonomi)
Varighet: [antall] uker

Kundens behov:
[2–3 setninger om problemet]

Løsning:
[2–3 setninger om hva teamet leverte]

Prosess:
- Uke 1–2: Innsikt og behovsavklaring
- Uke 3–6: Design og prototyping
- Uke 7–12: Utvikling og implementering
- Uke 13–15: Kvalitetssikring og overlevering

Resultat:
[2–3 setninger om hva som ble oppnådd]

Skjermbilder:
[visuelt materiale, anonymisert]
```

---

## 4. Studentenes egne porteføljesider

Hver student bør kunne vise frem *sitt* bidrag:
- Individuell side på portalen med studentens prosjekter
- Beskrivelse av studentens rolle og oppgaver
- Kodeutdrag / designskisser (med samtykke)
- Kobling til LinkedIn / GitHub / Behance
- Studenten kan lenke til porteføljen i jobbsøknader

---

## 5. Håndtering av prosjekter som ikke kan publiseres

| Årsak | Alternativ |
|-------|-----------|
| Kunden nekter samtykke | Studenten kan likevel beskrive prosjektet generisk uten å navngi kunden |
| Forretningshemmeligheter | Publiser kun prosess og metode, ikke spesifikke løsninger |
| Studenten nekter samtykke | Respekteres — fjern studentens navn fra case study |
| Dårlig kvalitet | Publiser ikke — bruk som intern læring |
